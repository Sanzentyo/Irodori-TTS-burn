# Layer-by-Layer Numerical Comparison

## Motivation

User observed that E2E audio from Rust and Python backends was "similar vibe but
not completely identical." This document records the results of an exhaustive
layer-by-layer comparison to determine whether the difference is:

1. **Expected** (different initial noise from different RNG implementations), or
2. **A bug** (systematic divergence at some layer).

## Methodology

### Small toy model (dim=64, 2 DiT layers)

`scripts/validate_numerics.py` was updated to register forward hooks on each
`DiffusionBlock` and on `in_proj`, capturing their outputs for comparison.

Captured tensors added to `target/validate_tensors.safetensors`:
- `after_in_proj` — output of `self.in_proj(x_t)` before any DiT block
- `block_0_out` — output of DiT block 0 (post-attention + post-MLP residual)
- `block_1_out` — output of DiT block 1

`src/model/dit.rs` was updated to add:
- `BlockDebugOutputs<B>` struct (public) with `after_in_proj` + `block_outputs`
- `forward_with_cond_debug()` method that captures per-block outputs
- Shared `forward_backbone()` private method eliminates code duplication between
  production path (`forward_with_cond_cached`) and debug path

`src/bin/validate.rs` was updated to:
- Call `forward_with_cond_debug()` instead of `forward_with_cond()`
- Compare per-block outputs against Python fixtures
- Add KV-cache consistency test (cached vs uncached forward)

### KV-cache consistency (Rust-internal invariant)

An additional test was added to `validate.rs` to verify that building a KV cache
and running `forward_with_cond_cached` gives the same result as `forward_with_cond`
(which does not use a cache). A failure here would indicate a bug in the
`build_kv_caches` / `forward_with_cond_cached` logic used in production inference.

## Results (2026-07-15)

All checks pass with excellent numerical precision:

| Check | max_abs_diff | Tolerance | Result |
|-------|-------------|-----------|--------|
| text_state (speaker) | 2.98e-7 | 1e-3 | ✓ PASS |
| speaker_state | 9.54e-7 | 1e-3 | ✓ PASS |
| after_in_proj | 0.00e+0 | 1e-3 | ✓ PASS |
| block_0_out | 5.96e-8 | 1e-3 | ✓ PASS |
| block_1_out | 5.96e-8 | 1e-3 | ✓ PASS |
| v_pred (speaker) | 0.00e+0 | 1e-3 | ✓ PASS |
| text_state (caption) | 4.77e-7 | 1e-3 | ✓ PASS |
| caption_state | 4.77e-7 | 1e-3 | ✓ PASS |
| v_pred (caption) | 0.00e+0 | 1e-3 | ✓ PASS |
| cached_vs_uncached | 0.00e+0 | 1e-5 | ✓ PASS |

**Conclusion**: The Rust implementation is numerically correct at every layer.
The audio difference between Python and Rust is entirely due to different initial
noise tensors generated by different RNG implementations (PyTorch Generator vs
Burn backend seed), **not an implementation bug**.

## Why audio sounds "similar but not identical"

- Python: `torch.Generator(device).manual_seed(seed)` → PyTorch's MT19937
- Rust: `B::seed(&device, seed)` → LibTorch backend's global RNG (different state)
- Both produce valid Gaussian noise; neither is "wrong"
- Different initial noise → different diffusion trajectory → different audio
- The content is equivalent (same text, same speaker) but the specific realization differs
- This is verified by the prior `--noise-file` experiment (earlier session) which showed
  only 2 latent frames of tail difference when using identical noise
